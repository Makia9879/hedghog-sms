# 短信发送者身份增强与平台归属回退 SOP

状态：已决定  
日期：2026-07-12  
适用范围：普通号码、106 短码/扩展码、虚拟号码、字母 Sender ID，以及容易被误称为“端口号”的长发送地址

## 结论

没有一个可供收件人 App 使用、能把任意短信发送者地址可靠映射到最终业务平台的权威通用查询接口。

- 监管部门的码号审批能说明某段号码资源的获批使用者或业务类型，但不等于实际使用该通道发送验证码的下游品牌。
- 云短信厂商要求企业申请/审核正文中的短信签名；这说明**正文签名**比终端显示的发送号码更接近业务身份。一个云短信/运营商通道可以服务许多企业。
- 普通手机号、虚拟号和长码可能被租用、转售、携号转网或回收；运营商/归属地查询最多说明号码资源，不证明当前平台。
- 106 长号码的前缀可提示码号类别或通道主体，尾部扩展通常是路由/客户扩展信息；不存在可假定完整、公开、长期稳定的“扩展码到品牌”表。
- 字母 Sender ID 是短信协议允许的发送地址形态，不是全球统一注册、由手机端加密验证的品牌证书。是否预注册、能否回复及防冒用强度取决于国家和运营商。
- SMS 的 originating address 不是面向 App 的端到端身份认证结果。伪造、错误配置、共享通道和号码复用都要求 App 把它当作特征，而不是平台事实。

因此 MVP **不内置“号码归属即平台”的规则，不自动调用第三方反查服务，不因一次外部查询结果训练模型**。优先级是：正文明确签名/品牌证据、本机已确认样本与本地模型、用户主动触发的候选查询、用户确认。

## 各类发送者能证明什么

| 类型 | 可获得信息 | 不能据此证明 | MVP 用法 |
| --- | --- | --- | --- |
| 普通手机号/E.164 长号 | 国家码、号码形态；合规服务可能给出当前运营商 | 当前使用企业、短信真实平台、号码未被回收/冒用 | HMAC 后作为模型特征；不展示“平台归属” |
| 106 短码/长扩展码 | 可查询到的码号持有/业务主体、号码层级 | 下游客户品牌；同一尾号永久归属同一平台 | 全地址和分层前缀分别作为弱特征；外部结果标“通道主体” |
| 其他短码/服务码 | 国家/运营商计划中的服务类型 | 最终内容平台；跨运营商含义一致 | 仅在有官方来源时显示服务类型 |
| 虚拟/租赁号码 | 号码格式、可能的服务商 | 租户身份、当前租期、之后不会复用 | 与普通号码同等弱证据 |
| 字母 Sender ID | 发件显示名字符串 | 全球唯一品牌所有权、端到端认证、未被冒用 | 规范化后作为强于随机长号但仍可冲突的特征 |
| 数据 SMS 端口 | `DATA_SMS_RECEIVED_ACTION` 的目标端口 | 普通短信列表中长号码的品牌身份 | MVP 不处理 data SMS；不得把 106 扩展码称作 Android 端口 |

## 中国短信通道现实

工信部现行《通信短信息服务管理规定》要求短信息服务经营许可和码号资源使用许可；《电信网码号资源管理办法》规定码号资源国家所有、审批使用。这提供了**码号层的合规主体**，没有建立面向收件人的最终品牌反查合同。

阿里云、腾讯云、华为云的官方短信文档都把“短信签名 + 正文”作为完整消息，并要求签名申请/审核或资质材料。签名可以表示企业、产品或业务；涉及第三方还可能需要授权。由此可得：

1. 已审核的正文签名是重要证据，但它仍是字符串，不是 App 能密码学验证的证书。
2. 云短信产品本身服务大量企业，通道号码映射到云厂商或基础运营商，不等于映射到验证码平台。
3. 对“106xxxx 属于某云厂商”的查询结果，界面必须写“通道/码号主体候选”，不能写“平台已确认”。

## 发送者规范化与本地特征

从系统 Provider 实时读取 `address`，在内存中规范化：

- Unicode NFC；去除仅用于显示的空白和常见分隔符，但保留原始数字/字母顺序。
- 纯数字/带 `+` 地址归为 numeric；只有能无歧义规范化时才补国家码，不能把所有短码补成 `+86`。
- `106...` 保留完整地址，同时派生长度和若干分层前缀；前缀只供模型学习，不在代码中硬编码“某平台”。
- 字母数字混合地址统一拉丁大小写，保留非 ASCII 字符并记录 `ALPHANUMERIC` 类型。
- null、空串、明显损坏地址统一为 `UNKNOWN`，不参与外部查询。

训练库不保存原始地址。按技术基线使用每安装 Keystore 密钥做 HMAC，生成：

- `sender_full`：完整规范化地址的 HMAC bucket；
- `sender_type`：numeric/short-code/alphanumeric/unknown；
- `sender_prefix`：适用时的分层前缀 HMAC bucket；
- 长度桶、是否 106 开头等非识别性结构特征。

这些特征与正文 HMAC n-gram 一起进入在线 MNB。每条 sender 特征计数最多 1，避免发送地址压倒正文。Keystore 密钥变化时，sender 特征与模型按既定 schema 重建。

## 本机发送者证据

App 可从**人工确认或经批准的自动确认训练样本**计算本机统计，但统计不是显式“发送者规则”：

- `confirmedCount`：该 sender HMAC 下已确认样本数；
- `distinctDays`：跨多少自然日；
- 各平台计数与 dominance；
- `conflict`：是否出现多个平台标签。

少于 2 条时不展示发送者历史推断。达到 3 条且跨至少 2 天，可在候选解释中显示“本机曾有 3 条确认属于 X”；只有达到至少 5 条、跨 3 天、主标签占比 `>=95%` 且无近期冲突，才称为“本机强佐证”。即使是强佐证，也只参与本地模型/阈值判定，不能跳过分类合同或生成永久 sender->platform 规则。

任一人工冲突会立即撤销“强佐证”状态；这正是共享通道的信号。标签重命名/合并按平台目录映射重新聚合，不能把别名当成冲突。

## 外部候选来源分级

### A：可引用的一手证据

- 工信部/通信管理局的码号审批、码号资源页面：只产出码号/通道主体 claim。
- 运营商官方查询或客服书面结果：只产出运营商、号码类型或通道主体 claim。
- 目标平台自己的安全/帮助页面明确列出的短信签名或发送地址：可产出平台候选；必须保存来源 URL、获取时间和适用范围。
- 云短信服务商登录后给其客户看的发送记录：仅号码所有者可查，不是收件人通用 API，不作为本 App 的默认数据源。

### B：弱候选

搜索引擎结果、LLM 回答、众包来电标记、论坛和商业反向号码库只能提供弱候选。它们可能过时、被投毒、把通道商当品牌或违反其服务条款。MVP 不把它们作为自动确认来源。

### MVP 取舍

MVP 不捆绑远程 sender lookup API。设置页可保留 `SenderLookupProvider` 替换接口，但默认实现仅提供“用户主动查询”：

1. 显示将离开 App/可能暴露发送地址的提示；
2. 用户确认后，才把规范化 sender 交给系统浏览器中的受信官方页面，或允许复制；
3. 返回 App 后由用户从已有平台或新平台中确认，不抓取浏览器页面、不自动导入结论。

后续若接入 API，接口必须返回结构化 claim，而非一个平台字符串：

```kotlin
data class SenderClaim(
    val subjectKind: SubjectKind, // CHANNEL_OWNER, CARRIER, PLATFORM_CANDIDATE
    val displayName: String,
    val evidenceUrl: String,
    val observedAt: Instant,
    val expiresAt: Instant?,
    val evidenceTier: EvidenceTier
)

interface SenderLookupProvider {
    suspend fun lookup(sender: NormalizedSender, consentToken: ConsentToken): List<SenderClaim>
}
```

没有本次显式 consent token 时 provider 必须拒绝网络请求。外部结果不直接调用分类器 `learn`。

## 正文无法判断时的回退 SOP

### 1. 本地解析

先执行验证码规则和正文签名/品牌提取。正文明确且与平台目录唯一匹配时，可按高精度正文规则归类；sender 只用于冲突检测。

### 2. 本地模型

正文不明确时，提取正文结构特征和 sender HMAC 特征，运行在线 MNB。展示最高候选、置信状态和可解释证据：正文片段类型、本机确认历史数量、是否存在 sender 冲突。绝不显示“号码官方属于 X”，除非确有一手 claim，且仍要标明 claim 类型。

### 3. 待标注

模型未达到用户选择的三档阈值，或 sender 历史出现多平台冲突，则进入“未知平台/待标注”。可以展示最多一个最高候选；其余只作解释，不自动创建标签。

### 4. 可选候选查询

用户点“查询发送者”后才进入外部流程。发送前展示精确 sender、目标域名和“可能泄露你正在使用某服务”的风险。只发送 sender，不发送正文、验证码、收件手机号、SIM/卡槽、短信 id、联系人或本机历史统计。

外部返回“通道主体”时单独展示，不能放进平台下拉框；只有来源明确声称目标平台的 claim 才能成为平台候选。过期、无 URL 或众包来源默认折叠为弱证据。

### 5. 确认与训练

- **用户确认/纠正**：立即给当前短信写人工标签，并把脱敏正文+sender 特征作为一个训练样本进入本地模型；这是 sender 查询结果进入训练的普通门槛。
- **本地自动归类**：只有完整本地分类器满足三档阈值时才给当前短信自动标签。sender 强佐证不能单独通过阈值；自动预测绝不回灌训练。
- **外部查询**：任何 sender lookup claim 都不能单独自动确认或训练。它必须由用户确认，或作为已经逐批批准的 LLM 辅助样本中的解释证据；但原始 sender 仍不得上传给 LLM。
- **已批准 LLM 自动确认模式**：遵守既有三工具 harness 和逐批批准合同；它可以按既定策略进入训练，但不能因 lookup claim 降低门槛，也不能把通道主体当成平台。

这样既保留自动分类能力，也避免“模型预测 -> 自动训练 -> sender 偏差不断放大”的闭环。

## 隐私与安全

发送者地址是通信元数据，能够暴露用户使用的银行、社交、医疗或工作服务。即便不发送正文，远程查询方仍能把 sender、IP、时间和 API 账户关联起来。

- 默认纯本地、默认不查询；不在后台扫描中调用网络。
- 不向通用搜索/LLM 自动上传原始 sender；单向哈希通常无法支持公网反查，也不等于匿名。
- 远程适配器必须 HTTPS、域名 allowlist、最小日志、短超时、无第三方分析 SDK；查询缓存只存 sender HMAC、claim、来源和过期时间。
- 不把查询 URL（含 sender 参数）、响应正文或原始 sender 写入日志、崩溃报告和备份。
- 用户可查看和清除所有外部 claim；清除 claim 不删除其已人工确认的本地训练样本，除非用户同时选择删除学习记录。
- 一手来源也会变化。平台/通道 claim 默认 90 天过期；平台官方页面明确长期声明可设 180 天，过期后只显示历史证据，不自动用于当前判断。

## 验收矩阵

至少构造以下脱敏样本：

1. 同一 106 长码发送两个不同平台验证码：必须产生 conflict，不能建 sender 规则。
2. 同一平台使用两个 106 扩展码和一个普通号：正文/模型仍可聚合到同一平台。
3. 数字短码、`+86` 普通号、字母 Sender ID、混合大小写和空地址规范化。
4. 106 扩展码与 data SMS port 不混淆；MVP 忽略 data SMS。
5. 2/3/5 条本机已确认样本的证据门槛，以及新增冲突后立即降级。
6. 外部 claim 为运营商、通道主体、平台候选和众包结果时，UI 类型与权限不同。
7. 未同意、离线、超时、证书失败、过期 claim 均回到本地待标注，不影响短信浏览。
8. 自动模型标签不增加训练计数；用户确认后恰好增加一个样本；纠正按既有模型事务回滚旧标签。
9. 网络抓包确认查询不含正文、验证码、收件号码、SIM/卡槽或短信 id；日志中无 sender。
