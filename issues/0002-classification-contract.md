## 问题

MVP 如何判定一条短信是验证码短信、如何定义和规范化“平台”、如何表达一条短信关联的卡槽与平台、如何处理一个发送方对应多个平台或一条短信无法确认平台，以及置信度、待确认和用户纠正遵循哪些确定性规则？

## 解决记录

已通过与用户逐题 grilling 解决：

1. **验证码判定**：仅当短信同时包含动态数字/字母码及验证码、校验码、OTP、verification code 等用途语义时自动判定；订单号、快递码、金额或普通数字不算，模糊项进入“待标注”。验证码判定可使用确定性规则。
2. **平台身份**：每条验证码短信最多一个主平台标签。正文中的平台/品牌证据优先于模型猜测；发送号码或签名只是特征，不能直接当作平台事实。无法确认时为“未知平台/待标注”。
3. **平台目录**：用户可维护平台名称，并执行重命名、合并和拆分。多候选只展示最高置信候选，其余仅作解释证据。
4. **人工标签**：人工标签永远覆盖自动结果。用户只需选择短信属于什么平台；该动作生成训练样本，不创建显式发送方规则。
5. **增量学习**：训练定义为完全端侧的轻量文本分类器增量学习，不是在手机微调大语言模型。原始正文只在特征提取时临时读取，训练库仅保存短信引用、脱敏特征、标签与必要训练元数据。
6. **更新范围**：新增或纠正样本只更新当前短信和本地模型，并用于后续新短信；不自动重新分类历史短信。
7. **置信交互**：提供“偏保守 / 平衡 / 偏自动”三个正式档位，各档位映射到内置阈值；低于阈值的结果进入“待标注”，不强行归入平台。MVP 不暴露原始数值参数。
8. **云端辅助标注**：默认关闭。用户可配置兼容 OpenAI 风格的 LLM API 提供商；每批上传前必须展示提供商、数量和脱敏预览并获得确认。App 移除验证码、手机号等敏感字段。受限 harness 只提供查看已批准样本、提交答案和训练本地模型三类动作；确认可设人工或自动档位，自动档位允许创建新标签。API Key 存 Android Keystore。
9. **标签合并**：用户可将重复平台标签及其历史短信合并到一个主标签；旧标签保留为别名，合并结果用于后续训练，但不执行模型遗忘。

验收必须覆盖明显验证码、普通数字误报、未知平台、同一发送通道服务多平台、人工覆盖、三个置信档位、本地增量学习不改历史结果、标签合并，以及云端模式默认关闭、人工/自动确认和逐批上传批准。
